# 一、先分析整个游戏

我会先站在"比赛设计者"角度分析。

## 这是一个什么游戏？

一句话：

> **带资源管理的双人实时物流对抗。**

它融合了：

- 路径规划
- 背包管理
- 资源抢夺
- 实时博弈
- 风险控制
- 对手预测

有一点像：

> 欧卡运输 + 炉石猜拳 + 皇室战争抢资源 + 星际争霸运营

------

# 二、真正的目标是什么？

很多人第一眼会觉得：

> 尽快送到终点。

实际上不是。

真正目标：

最终得分最大：

```
最终分 =
交付时间
+
剩余好果
+
剩余鲜度
+
任务分
+
悬赏分
-
惩罚
```

因此：

**送得快 ≠ 赢**

例如：

你：

```
200帧到终点
剩40果
鲜度30
```

别人：

```
260帧到终点
剩85果
鲜度90
任务拿满
```

别人可能赢很多。

所以：

> 这是一个典型 Multi-objective Optimization（多目标优化）。

# 四、地图分析

地图固定只有：

```
S01~S15
```

最多15个点。

意味着：

**非常小。**

所以：

完全没必要用复杂导航。

建议：

开局直接：

```
Floyd

或者

Dijkstra
```

全部预计算。

得到：

```
dist[i][j]
next[i][j]
```

以后：

任何路径：

O(1)

得到下一步。

# 十二、推荐整体架构（无AI）

建议采用如下模块化设计：

```
Client
│
├── Network
│
├── GameState
│
├── Map
│
├── PathPlanner
│
├── WeatherManager
│
├── TaskManager
│
├── ResourceManager
│
├── SquadManager
│
├── WindowManager
│
├── EnemyTracker
│
├── Evaluator（动作评分）
│
├── Strategy
│
└── Agent
```

其中：

- **GameState**：维护所有公开状态（位置、资源、天气、任务等）。
- **PathPlanner**：预计算最短路与路径代价。
- **EnemyTracker**：记录对手位置、资源、历史行为和窗口出牌统计。
- **Evaluator**：对所有候选动作进行统一评分。
- **Strategy**：根据比赛阶段调整各项评分权重。
- **Agent**：每帧生成候选动作 → 评分 → 输出得分最高的合法动作。

------

# 十三、建议采用三阶段策略

根据比赛规则，整个对局可以划分为三个阶段，不同阶段使用不同权重，而不是一套策略打完整局。

| 阶段 | 帧数（建议） | 核心目标                                                     |
| ---- | ------------ | ------------------------------------------------------------ |
| 前期 | 1～200       | 收集高价值资源、完成高 ROI 皇榜任务、规划路线                |
| 中期 | 200～450     | 根据天气和敌方位置调整路线，争夺关键资源与任务，控制鲜度损耗 |
| 后期 | 450～600     | 尽快完成验核和交付，必要时使用终局急策，避免因贪任务导致无法交付 |

策略切换只需要依据当前帧数、剩余鲜度、距离终点和双方分数预估进行判断即可，不需要任何机器学习。

------

# 十四、初步结论

我认为，这个比赛真正考察的不是 AI，而是**算法工程能力**。由于地图规模很小（15 个站点）、规则完全公开、每帧需要快速响应，因此**最优方案不是使用 LLM 或深度学习，而是一个高性能、可解释的规则决策系统**。

如果让我从零开始设计一个有竞争力的 Agent，我会采用以下技术组合：

- **路径规划**：Floyd 或 Dijkstra 预计算最短路，并基于时间、鲜度、天气和风险构建综合代价。
- **决策框架**：Utility AI（动作评分系统），统一评估所有合法动作并选择最高分。
- **状态管理**：有限状态机（FSM）管理不同比赛阶段和处理流程。
- **敌方建模**：记录历史行为、窗口出牌统计、资源获取情况，进行轻量预测。
- **规则优化**：为任务、资源、设卡、清障、终局急策等设计启发式评分函数，而不是大量 `if-else`。
- **性能目标**：所有决策保持毫秒级完成，确保不会因客户端与服务器通信时延导致超时。

**这是目前这类规则型 Bot 比赛中最稳定、最容易迭代优化，也是最有竞争力的技术路线。**
